# 输出模板

## 0. 正式写作前的确认模版

收到用户背景后，先给出类似下面的确认块，再等待用户选项：

```markdown
我先给你 3 个可执行方向，你确认后我再正式写：

1. 完整商业化方案版
- 适用：需要正式汇报文档、项目方案、课件底稿。
- 我会怎么写：按“纠偏 -> STAR -> 背景 -> 目标 -> 方案 -> 结果 -> 推进策略 -> 结论”展开。
- 产出：一版可直接用于汇报或继续改 PPT 的长文方案。

2. STAR 面试复盘版
- 适用：需要讲项目经历、面试话术、晋升答辩。
- 我会怎么写：重点压缩在业务问题、产品判断、关键动作和结果。
- 产出：一版 800-1500 字的 STAR 叙述。

3. 改写润色版
- 适用：你已有初稿，但现在更像算法稿/功能稿。
- 我会怎么写：保留事实和数据，重写标题、结构、语言和结论。
- 产出：一版 PM 视角重写稿。

推荐：`[填写最适合当前材料的方案]`
原因：`[用 1-2 句解释为什么]`
```

## 1. 默认完整结构

不确定用什么骨架时，优先使用下面这套结构，再按场景裁剪：

```markdown
# [项目名]商业化方案（产品经理视角）

## 一、先纠偏：这不是[技术/功能]，而是[业务机会/平台能力]
[先用 1-3 段把项目重新定义清楚]

## 二、正确的 STAR 写法
### S：Situation
[业务现状、痛点、风险]
### T：Task
[目标、边界、约束]
### A：Action
[3-5 个平台化动作]
### R：Result
[线上价值、产品化解读、下一步]

## 三、项目背景
### 3.1 商业问题
### 3.2 为什么现在必须做
### 3.3 当前验证结论 / 市场洞察

## 四、项目目标
### 4.1 北极星指标
### 4.2 过程指标
### 4.3 效率指标
### 4.4 风险指标
### 4.5 分阶段目标

## 五、核心方案设计
### 5.1 方案定位：不是[X]，而是[Y]
### 5.2 能力/机制拆解
### 5.3 为什么这条路线对 PM 成立

## 六、实验结果或业务测算的产品化解读
### 6.1 整体结论
### 6.2 分模块 / 分资源位 / 分场景解读
### 6.3 为什么未起量 / 当前限制在哪里

## 七、项目动作
### 7.1 [动作一]
### 7.2 [动作二]
### 7.3 [动作三]

## 八、分阶段推进策略
### 阶段 1
### 阶段 2
### 阶段 3

## 九、面试或汇报时可以怎么讲
[一段 150-250 字口播稿]

## 十、最终结论
[最多 3 点]
```

## 2. 压缩版 STAR

当用户只要面试话术、项目复盘或一页式回答时，压缩为：

1. `先纠偏`
说明项目真正解决的商业问题，不要从技术名词开场。
2. `S/T/A/R`
每部分 1-2 段，重点放在业务机会、产品判断和结果。
3. `补一段 PM 视角总结`
明确为什么这个项目体现的是平台能力和商业化价值。

## 3. 改写现有材料时的处理顺序

1. 保留原始事实、数据、实验结论。
2. 重写标题和开场，让它先像商业化方案。
3. 把技术描述改写成产品能力、机制设计、风险边界和放量策略。
4. 把结果重写成业务解读，不只列数字。
5. 用一句总结束把项目升级成“值得立项/可扩量/可复制”的平台能力。

## 4. 缺少结果时怎么写

如果用户还没有线上实验结果，不要硬编“已验证”。改为：

- 给出业务测算或方向性收益预期。
- 写清验证方案、灰度计划和关键护栏。
- 把“结果解读”替换成“验证路径与预期判断”。
