# 美团闪购店内推荐 TAB 召回-排序-扶持一体化升级方案

适用角色：美团闪购 O2O 商业化策略产品经理  
写作方式：延续《80期商业化5组-淘宝闪购广告新品扶持》在第 4 部分之后的骨架，但内容完全按“店内推荐 TAB”场景重写。


# 4. 项目背景

## 4.1 业务现状与问题诊断

店内推荐 TAB 是用户进店后的核心承接位之一，承担“继续逛店、补充决策、带动加购”的职责。当前该场域并不是典型广告位，但它本质上也是平台重要的流量调度位，既影响用户体验，也影响店内成交效率和后续流量扶持资源的利用效率。

过去 30 天抽样看，店内推荐 TAB 的基础盘子如下：

- 日均进入用户 UV 约 210 万。
- 日均推荐请求约 780 万。
- 日均商品卡曝光约 4680 万。
- 当前 CTR 约 8.6%。
- 曝光到加购率约 2.05%。
- 点击到支付转化率约 3.95%。
- 客单价约 52 元。
- 对应日均承接 GMV 约 826 万。

公式如下：

`GMV = 曝光 * CTR * 点击支付转化率 * 客单价`

代入基线：

`4680万 * 8.6% * 3.95% * 52 ≈ 826万/日`

现阶段链路的核心问题集中在三类。

### 1、召回同质化严重，用户“越刷越像”

当前推荐 TAB 主要依赖三类候选：

- 固定热销池，占召回曝光的 44% 左右。
- 基础 I2I 相似商品召回，占 36% 左右。
- 运营规则池，占 20% 左右。

这套方式能保证“有货可推”，但不够理解用户当下会话。问题表现为：

- 会话内 Top12 卡片中，同品牌重复曝光占比 41%。
- 会话内 Top12 卡片中，同规格或同价带重复曝光占比 57%。
- 第一屏 6 卡中，至少有 4 卡高度相似的请求占比达到 33%。
- 卡位越往后，点击衰减越快，1-3 卡 CTR 为 9.4%，4-6 卡 CTR 下降到 7.1%。

这说明当前 I2I 更像“给用户看同款”，而不是“帮用户继续逛店”。当用户已经看过一瓶 550ml 可乐后，再连续推荐 500ml、600ml、同品牌无糖款，系统虽然维持了相关性，但会显著消耗探索感和新鲜感。

### 2、离线产池更新慢、适配弱，底池“分高但不适合逛”

当前离线池以 T+1 产出为主，策略是“按综合分取 topN”，缺少对实时供给变化和集合覆盖的考虑，导致三个问题：

- 单店平均可售 SKU 约 620 个，但当前推荐底池平均只有 148 个，供给覆盖率仅 23.9%。
- 价格变动、库存波动、店内热卖迁移主要依赖次日更新，热点迁移滞后约 18-24 小时。
- 抽样看，进入产池的 SKU 中有 11.8% 在当天出现“缺货、价格大变动、转化骤降”问题，但仍可能留在候选中。

结果是：候选虽然分数不低，但未必适合“逛店场景”。用户真实感知是“怎么老是那几个货”“怎么刚点进去就没货”“怎么店里热卖现在才被推上来”。

### 3、流量扶持能力粗放，资源投了但未必产生增量

业务侧希望对不同 query、不同 SPU 做更精细的流量扶持，例如：

- 用户从“可乐”“薯片”“早餐面包”等强意图 query 进店后，希望重点扶持相关 SPU。
- 对新品、活动品、平台策略品，希望在不伤体验的前提下获得确定性试流量。
- 对不同类目，希望能灵活配置扶持强度，而不是统一 boost。

但当前扶持方式更多是“统一加权”，缺少三类能力：

- 缺少 query 维度的精细映射，很多扶持是“泛加权”，不是“按需求扶持”。
- 缺少自然结果保护，强 boost 时容易出现自然好结果被过度挤压。
- 缺少增量评估，扶持对象未必真的带来新增点击和新增成交。

抽样看：

- 当前扶持曝光中，有 37% 落在低增量 query 或弱相关 SPU 上。
- 当扶持强度拉高时，自然结果 CTR 平均下滑 0.18pp~0.22pp。
- 当前扶持流量的千次曝光增量 GMV 仅约 17 元，说明资源利用效率偏低。

### 4、结论

这个问题不是单点优化某个召回通路就能解决，而是要把“召回、排序、扶持”统一成一条链路来重做：

- 召回侧解决“能不能召回到更懂会话的候选”。
- 产池侧解决“底池够不够丰富、够不够新鲜、够不够适合逛”。
- 排序侧解决“相关性、转化效率、多样性和扶持如何平衡”。
- 扶持侧解决“资源是否按 query 和增量价值投放，而不是粗放加权”。


## 4.2 流量空间探查

### 1、为什么要做流量空间探查

店内推荐 TAB 不是一个边缘功能，而是店内自然分发里最适合做“承接优化 + 经营调度”的位置。若不先确认它的流量结构、query 分布和供给空间，就容易把项目做成算法优化题，而不是一个可规模复制的策略产品项目。

### 2、用户需求并不单一，query 和会话意图都值得经营

按近 30 天请求拆分，店内推荐 TAB 的请求来源大致如下：

- 带 query 进店或带 query 上下文的请求：46%。
- 从店内搜索结果切换到推荐 TAB 的请求：17%。
- 从商品详情页继续下滑或回流到店内推荐的请求：22%。
- 从店铺首页、类目导航、活动页进入的泛逛店请求：15%。

进一步看 GMV 贡献：

- 带明确 query 的请求贡献了约 39% GMV。
- 从商品详情页回流的会话贡献了约 24% GMV。
- 也就是说，“query + 会话链路”两类请求合计贡献了约 63% 的店内推荐 TAB GMV。

这意味着：当前 TAB 最大的问题不是没流量，而是没有把强意图 query 和会话上下文充分利用起来。

### 3、供给空间明显不足，当前底池无法支撑“逛店”

抽样 5000 家活跃门店后发现：

- 平均单店可售 SKU 为 620 个，但当前离线底池平均仅覆盖 148 个。
- 高流量门店中，可售 SKU 超过 1000 个的门店占比 31%，但推荐底池仍普遍卡在 180 个以内。
- 当前 Top12 卡片的平均品牌数仅 4.3 个，价格带数仅 2.7 个，说明多样性不足。
- 当前候选中缺货或弱可售商品曝光率约 0.9%，仍有明显优化空间。

从“逛店体验”视角看，店内推荐应该天然承担“补充选择”和“扩充篮子”的职责，而不是只做“同款替代”。

### 4、流量扶持也有真实空间，但必须建立精控机制

在业务调研和历史活动复盘中，可以看到两类真实需求：

- 运营和商家希望围绕重点 query 扶持目标 SPU，例如“气泡水”“吐司面包”“洁面巾”等。
- 平台希望对新品、活动品、策略品建立可控扶持，不再依赖拍脑袋加权。

对高意图 query 做抽样后发现：

- 可配置的高价值 query 包约 6200 个，覆盖 29% 的店内推荐请求。
- 这些 query 对应请求的 CTR 比泛流量高出 18% 左右，点击后支付转化率高出 11% 左右。
- 若能把扶持从“泛 boost”升级为“by query 的增量扶持”，理论上扶持千次曝光增量 GMV 可从 17 元提升到 27-31 元。

### 5、初步结论

店内推荐 TAB 同时具备三种空间：

- 有体验优化空间：同质化明显，用户逛店效率仍可提升。
- 有供给优化空间：当前底池覆盖率和新鲜度都不足。
- 有经营调度空间：query 和 SPU 的扶持需求真实存在，但缺少可控机制。

因此，这个项目值得按“召回-排序-扶持一体化”立项，而不是拆成若干零散优化点。


# 5. 项目目标

## 5.1 指标定义

### 北极星指标

- 店内推荐 TAB 日均承接 GMV。

### 召回与供给过程指标

- 会话级有效召回覆盖率。
- 离线产池供给覆盖率。
- 产池新鲜度命中率。
- 缺货/下架过滤命中率。
- Top12 多样性指标（品牌数、价格带数、同质重复率）。

### 效率过程指标

- CTR。
- 曝光到加购率。
- 点击到支付转化率。
- GMV/千次曝光。
- 扶持千次曝光增量 GMV。

### 生态风险指标

- 自然结果 CTR 跌幅。
- 自然结果 GMV/千次曝光跌幅。
- 扶持曝光占比是否超过红线。
- 缺货曝光率。
- 用户重复曝光率。

### 经营效率指标

- 扶持正增量命中率。
- 配置生效率。
- query-SPU 配置覆盖率。
- 运营策略回滚时效。


## 5.2 目标测算

本项目一期先覆盖超市便利、酒饮、休食、个护四类高逛店类目，约占店内推荐 TAB 请求的 46%。

### 1、基线盘子

- 一期范围日均曝光：`4680万 * 46% = 2153万`
- 基线 CTR：8.6%
- 基线点击支付转化率：3.95%
- 基线客单价：52 元

因此一期范围基线 GMV 为：

`2153万 * 8.6% * 3.95% * 52 ≈ 380万/日`

### 2、一期目标区间

项目上线后，希望通过召回、产池、排序、扶持四层联动，实现以下保守目标：

- CTR 提升至 8.95%~9.10%。
- 点击支付转化率提升至 4.05%~4.10%。
- 客单价提升至 52.3~52.5 元。

下限测算：

`2153万 * 8.95% * 4.05% * 52.3 ≈ 408万/日`

上限测算：

`2153万 * 9.10% * 4.10% * 52.5 ≈ 420万/日`

输出结果：

- 一期范围日均新增 GMV 约 `28万~41万`。
- 对应 GMV 提升约 `+7.3%~+10.9%`。

### 3、全量 eligible 流量预估

若后续扩到全部符合条件的店内推荐流量，且效果保持不变：

- 全量日均 GMV 提升约 `60万~90万`。
- 对应日均新增订单约 `1.1万~1.6万单`。

### 4、为什么这个测算值可信

这个预估是偏保守的，原因有三点：

- 一期只覆盖四个高逛店类目，没有纳入鲜花、母婴、宠物等可扩展类目。
- 扶持只按高价值 query 白名单做，不把全部 query 一次性开放。
- 排序保护对自然体验设置了明确红线，不会为了短期拉升强推扶持流量。


# 6. 项目动作

## 6.1 动作一：在线 I2I 召回升级，让推荐“懂会话”而不是只懂当前商品

### 6.1.1 目标

解决当前 I2I 只会围绕单商品做相似扩展的问题，让召回结果不再只是“同款堆叠”，而是能识别用户当前在店内到底是补货、凑单、尝新还是比较决策。

### 6.1.2 核心输入信号

在线召回请求引入以下四类信号：

- query 信号：用户进店 query、店内搜索 query、来源场域词包。
- 会话信号：最近 5 个浏览/点击/加购商品序列。
- 商品信号：当前浏览商品的类目、品牌、规格、价格带、活动状态。
- 店内实时信号：近 30 分钟热销迁移、库存变化、价格波动、活动变化。

系统将这些信号统一加工成 `IntentProfile`：

- 主需求类目。
- 价格带偏好。
- 品牌偏好强弱。
- 会话阶段标签（补货、凑单、尝新、对比）。

### 6.1.3 多路召回结构

由单一路 I2I 升级为五路召回并行：

#### 1、R1：同需求替代召回

- 目标：保障相关性，不让用户“逛着逛着跑偏”。
- 来源：类目、口味、规格、价格带相近的语义相似商品。
- 拉回数量：Top80。

#### 2、R2：关联购召回

- 目标：服务补货和凑单，提升跨品类加购。
- 来源：购物篮共现、订单共购、同店连带购买关系。
- 拉回数量：Top60。

#### 3、R3：实时热销迁移召回

- 目标：及时承接店内临时热卖和活动爆品。
- 来源：近 30 分钟订单、点击、加购热度变化。
- 拉回数量：Top50。

#### 4、R4：query 意图召回

- 目标：承接带 query 会话，不让用户进店后丢失原始需求。
- 来源：query 与 SPU 的映射词包、类目约束、强相关货品集合。
- 拉回数量：Top40。

#### 5、R5：探索补充召回

- 目标：解决“越刷越像”，强制保留适度探索空间。
- 来源：新品牌、新价格带、新规格、同类高点击新货。
- 拉回数量：Top40。

多路召回合并后，经过去重和过滤，最终保留 180-220 个有效候选进入排序。

### 6.1.4 合并规则与护栏

为避免召回层就把候选做得过窄，在合并阶段增加三类约束：

- 同品牌候选在 Top50 中占比不超过 35%。
- 同价格带候选占比不超过 50%。
- 关联购 + 探索类候选合计占比不低于 30%。

对于无明确 query、无明显序列信号的冷会话，退回“店级热销 + 店内关联购 + 类目探索”的保守组合，避免空召回。

### 6.1.5 新增能力点

- 推荐请求透传 query 与会话序列。
- 实时热销迁移服务接入召回。
- 候选打标输出召回来源，为后续排序解释和“应出未出”归因做准备。

### 6.1.6 预期收益

- 会话级有效召回覆盖率提升 `+12pp~+16pp`。
- 同品牌连续曝光占比从 `41%` 降到 `26%~30%`。
- 单独贡献 CTR 提升约 `+0.15pp~+0.22pp`。


## 6.2 动作二：离线产池重构，把底池从“按分取 topN”升级为“可逛、可售、可更新”

### 6.2.1 目标

解决离线池更新慢、覆盖窄、对实时供给变化不敏感的问题，让底池既有转化能力，也能支持逛店多样性和实时经营。

### 6.2.2 新产池结构

按店铺规模分层产池：

- S 店：日均 TAB 请求 < 200，底池 180 个。
- M 店：日均 TAB 请求 200-1000，底池 300 个。
- L 店：日均 TAB 请求 > 1000，底池 500 个。

每个底池再拆成三层：

- 转化层 40%：稳定高转化、高点击、强相关商品。
- 探索层 35%：新品牌、新规格、新价格带、关联购候选。
- 经营层 25%：新品、活动品、策略扶持品、利润健康品。

这意味着底池不再只看单点分数，而是同时考虑“好卖”“可逛”“可经营”。

### 6.2.3 产池打分逻辑

定义离线选池分：

`PoolScore = 0.35*销售热度 + 0.25*转化质量 + 0.15*供给稳定 + 0.15*覆盖增益 + 0.10*经营价值`

其中：

- 销售热度：近 7 天点击、加购、成交综合表现。
- 转化质量：CTR、加购率、点击支付转化率。
- 供给稳定：库存稳定性、价格稳定性、履约稳定性。
- 覆盖增益：补充品牌、价格带、规格和类目空缺的能力。
- 经营价值：是否为新品、活动品、重点经营 SPU。

在 topN 选取时增加覆盖约束，不再允许高分商品把一个品牌或一个价格带全部占满。

### 6.2.4 更新机制升级

离线产池从单一 T+1 升级为“三层刷新”：

- T+1 全量重建：保证全局一致口径。
- 小时级增量刷新：对高流量门店每小时刷新一次。
- 事件级准实时刷新：库存、价格、上下架、活动切换发生变化时，10 分钟内完成差量更新。

### 6.2.5 过滤与归因能力

产池输出前新增四类过滤：

- 缺货或库存深度不足。
- 价格在 24 小时内波动过大。
- 近期投诉、履约或售后风险异常。
- 店内无图、无标题、无有效售卖信息。

同时上线“应出未出”归因：

- 未进入产池。
- 进入产池但实时过滤。
- 进入候选但排序靠后。
- 进入排序但被重排打散替换。

这样后续无论是算法优化还是运营复盘，都能明确问题到底卡在哪一层。

### 6.2.6 预期收益

- 单店底池平均覆盖率从 `23.9%` 提升到 `41%~48%`。
- 产池新鲜度命中率从 `88.2%` 提升到 `96%+`。
- 缺货曝光率从 `0.9%` 降到 `0.3%~0.4%`。
- 单独贡献 CTR 提升约 `+0.10pp~+0.15pp`。


## 6.3 动作三：排序与重排升级，在相关性、效率、多样性和扶持之间做统一平衡

### 6.3.1 目标

解决当前排序过度依赖基础规则、无法兼顾“转化效率”和“逛店体验”的问题，同时给扶持流量留出明确、可控、不过载的空间。

### 6.3.2 排序结构

排序链路分三段：

- 粗排：200 候选到 80 候选，快速过滤低相关、低可售、弱质量商品。
- 精排：80 候选到 30 候选，重点计算成交概率和经营价值。
- 重排：30 候选到最终 12/24 卡位，负责多样性、打散、扶持保护和自然结果护栏。

### 6.3.3 统一排序分

定义基础排序分：

`Score_base = pCTR * pATC * pCVR * SupplyFactor * PriceFactor`

在此基础上叠加两个增量项：

`Score_final = Score_base + λ1*ExploreGain + λ2*SupportUplift - λ3*HomogeneousPenalty`

其中：

- `ExploreGain`：补足新品牌、新规格、新价格带、新关联品时的收益。
- `SupportUplift`：query-SPU 扶持带来的预估增量收益，不是简单 boost。
- `HomogeneousPenalty`：同品牌、同规格、同价带连续堆叠时的惩罚项。

不同会话下的权重动态调整：

- 强 query 会话：更重相关性和转化效率。
- 商品详情回流会话：更重关联购和互补品。
- 泛逛店会话：更重探索收益和多样性。

### 6.3.4 Top 区重排规则

在第一屏和前两屏增加明确护栏：

#### 第一屏 Top6

- 至少包含 3 个品牌。
- 同品牌不超过 2 个。
- 同 SPU 仅允许 1 个。
- 同价格带不超过 3 个。
- 至少保留 1 个关联购或探索位。
- 扶持卡位最多 1 个。

#### 前两屏 Top12

- 扶持卡位最多 2 个。
- 不允许两个扶持卡位相邻。
- 若扶持插入导致基础排序总分损失超过 8%，则放弃插入。

### 6.3.5 自然结果保护机制

扶持和探索都必须服从自然体验护栏：

- 自然 CTR 跌幅红线：不超过 0.15pp。
- 自然 GMV/千次曝光跌幅红线：不超过 2%。
- 扶持曝光占比红线：Top12 中不超过 16.7%，即最多 2 卡。

一旦连续两天触线，自动降级为“只做召回、不做排序 boost”。

### 6.3.6 预期收益

- 曝光到加购率提升 `+0.12pp~+0.18pp`。
- 点击支付转化率提升 `+0.08pp~+0.12pp`。
- Top12 品牌数从 `4.3` 提升到 `6.0~6.5`。
- 单独贡献 CTR 提升约 `+0.10pp~+0.15pp`。


## 6.4 动作四：by query 的 SPU 扶持配置化，做“可控、可解释、看增量”的流量扶持

### 6.4.1 目标

解决当前扶持只能统一加权、难以按 query 精准调度的问题，让扶持从“拍脑袋加分”升级为“围绕需求做增量经营”。

### 6.4.2 query 分层

将可扶持 query 拆为四层：

- Q1：强商品词，如“可乐 330ml”“吐司面包”。
- Q2：品牌词或品牌品类词，如“元气森林气泡水”“三只松鼠零食”。
- Q3：场景词，如“早餐”“夜宵”“办公室零食”。
- Q4：泛类目词，如“饮料”“饼干”“洗护”。

不同层级的扶持强度不同：

- Q1、Q2 允许更高扶持精度，但前提是 query-SPU 强相关。
- Q3 主要扶持场景匹配和关联购。
- Q4 只做轻扶持，不做强插前排。

### 6.4.3 SPU 准入规则

只有满足以下条件的 SPU 才能进入扶持池：

- query-SPU 相关性分不低于 0.75。
- 库存深度大于 1.5 天，且近 24 小时可售稳定。
- 价格竞争力处于类目中位数的 `-15% ~ +20%` 区间内。
- 近 7 天履约和售后风险不高于类目均值的 1.2 倍。
- 自然表现不低于类目 P30，或能用同品牌相似品做先验补齐。

不满足条件的商品不能因为“想扶”而强行进入。

### 6.4.4 扶持等级设计

为避免一刀切 boost，扶持拆成三个等级：

#### L1：召回扶持

- 只负责把目标 SPU 稳定召回到候选池。
- 不直接改排序。
- 适用于新品试流量、弱数据商品。

#### L2：轻排序扶持

- 在重排阶段给予轻度 `SupportUplift`。
- 只在 query 强相关、自然结果未受损时生效。
- 适用于高潜力目标品。

#### L3：确定性扶持

- 仅对白名单活动或核心策略品开放。
- 可以保证 query bucket 内最小曝光份额，但仍受 Top 区卡位上限和自然红线约束。
- 适用于大促、重点经营品、平台策略专项。

### 6.4.5 扶持增量判定逻辑

扶持不再看“有没有曝光”，而是看“有没有新增价值”。

定义增量分：

`UpliftScore = 预估新增点击 * 预估支付转化率 * 客单价 - 自然结果挤压成本`

只有当 `UpliftScore > 0` 时，扶持才继续保留；否则自动降级或退出。

增量验证采用两种方式：

- AB 主实验：观察整体 CTR、GMV、自然跌幅。
- query bucket 周期开关实验：验证具体 query-SPU 对是否真的有增量。

### 6.4.6 配置后台能力

新增策略配置后台，支持业务侧配置：

- query 包。
- SPU 列表。
- 生效门店/城市/类目范围。
- 扶持等级。
- 起止时间。
- 曝光目标或订单目标。
- 红线阈值和自动回滚规则。

同时输出统一看板：

- 每个 query 包的扶持曝光。
- 每个 SPU 的增量点击、增量 GMV。
- 自然结果挤压情况。
- 触线与回滚记录。

### 6.4.7 预期收益

- 扶持正增量命中率从 `54%` 提升到 `72%~78%`。
- 扶持千次曝光增量 GMV 从 `17元` 提升到 `27~31元`。
- 低增量扶持曝光占比从 `37%` 降到 `15%~20%`。


## 6.5 动作五：策略平台化与监控闭环，形成统一流量生态调度能力

### 6.5.1 目标

防止召回、排序、扶持各自为政，建立一套能配置、能监控、能回滚、能复盘的统一策略平台。

### 6.5.2 平台能力设计

统一流量调度台负责管理三层能力：

- 召回层：多路召回开关、候选配额、召回优先级。
- 排序层：多样性阈值、Top 区规则、自然结果护栏。
- 扶持层：query-SPU 配置、扶持等级、回滚规则。

### 6.5.3 核心监控面板

上线三类核心看板：

#### 1、链路看板

- 召回覆盖率。
- 产池覆盖率。
- 新鲜度命中率。
- 缺货过滤率。

#### 2、体验看板

- 同质化指数。
- Top12 品牌数。
- 自然 CTR 跌幅。
- 重复曝光率。

#### 3、经营看板

- query-SPU 扶持覆盖率。
- 扶持正增量命中率。
- 扶持千次曝光增量 GMV。
- 自动降级/回滚次数。

### 6.5.4 回滚与兜底

若出现以下情况，系统自动退回现网保守策略：

- 自然 CTR 连续两天跌幅超过 0.15pp。
- 缺货曝光率超过 0.5%。
- 某 query 包扶持千次曝光增量 GMV 连续三天为负。
- 某类目出现明显同质化恶化或客诉异常。

### 6.5.5 预期收益

- 运营策略配置时效从 `T+1` 降到 `2小时内`。
- 策略问题定位由“人工排查半天以上”降到“15 分钟内定位到层级”。
- 复盘可以从“只看到结果波动”升级为“能归因到召回、产池、排序、扶持任一环节”。

# 9. 项目复盘

## 9.1 项目 STAR

### S 背景

店内推荐 TAB 原有策略以固定热销池、基础 I2I 和简单规则排序为主，已无法适应当前店内供给丰富和经营调度精细化的要求。用户侧感知是同质化严重、逛店疲劳；业务侧感知是底池更新慢、扶持粗放，资源利用效率不高。

### T 目标

在不突破自然体验红线的前提下，完成店内推荐 TAB 的召回、排序、扶持一体化升级，提升店内推荐承接 GMV，并建立可配置、可解释、可复盘的 query-SPU 流量扶持能力。

### A 动作

1. 在线 I2I 召回升级：从“围绕当前商品做相似召回”升级为“围绕 query 和会话做多路召回”。
2. 离线产池重构：从 T+1 topN 产池升级为“分层底池 + 小时级刷新 + 事件级更新”。
3. 排序与重排升级：统一相关性、效率、多样性和扶持目标，并增加自然护栏。
4. by query 的 SPU 扶持配置化：把扶持从统一 boost 升级为按 query 精准调度、按增量保留。
5. 策略平台化与监控闭环：建立配置台、归因打点、自动回滚和统一看板。

### R 结果

按一期范围测算，预计可带来：

- CTR 提升 `+0.35pp~+0.50pp`
- GMV 提升 `+7.3%~+10.9%`
- 日均新增 GMV `+28万~+41万`
- 扶持千次曝光增量 GMV 提升 `+10元~+14元`

若全量 eligible 流量推开，预计可贡献：

- 日均新增 GMV `+60万~+90万`
- 日均新增订单 `+1.1万~+1.6万单`

## 9.2 后续优化点

### 优化点 1：扶持从规则引擎升级为 uplift model

当前扶持是否保留，仍以规则和简化增量分为主。后续可以引入 uplift model，直接预测“不给扶持”和“给扶持”两种情况下的差值，使资源更聚焦真正有增量的 SPU。

### 优化点 2：会话理解从规则标签升级为实时意图状态机

当前 `IntentProfile` 仍偏规则化。后续可将补货、凑单、尝新、比价等状态做成实时会话状态机，让召回和排序对用户阶段更敏感。

### 优化点 3：扶持配置向运营自助开放，但必须带护栏

后续可逐步把 query 包和 SPU 包配置能力开放给行业运营，提高策略配置效率；但必须绑定库存、质量、自然红线和回滚机制，避免“配置权下放 = 风险下放”。

### 优化点 4：建立店内推荐“应出未出”标准报表

把“召回不到、产池未进、实时过滤、排序靠后、重排替换”做成标准日报，帮助业务、算法、运营在一个口径上讨论问题，减少低效对齐成本。
