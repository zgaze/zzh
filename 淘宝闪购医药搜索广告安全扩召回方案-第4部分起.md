# 淘宝闪购医药搜索广告安全扩召回方案（第4部分起）

说明：以下内容按《80期商业化5组-淘宝闪购广告新品扶持》的写法展开，默认这是一个完整项目方案文档中的第 4-9 部分；其中第 8 部分结果按“两周灰度验收”的合理区间给出，便于直接用于方案汇报。

## 4. 项目背景

### 4.1 商家需求调研结果

本期围绕淘宝闪购 O2O 医药搜索广告场景，访谈了 13 家连锁药店、5 个医药品牌、4 位行业运营同学，核心结论如下：

#### 1、新增预算真实存在，但今天主要卡在“搜不到可投 query”

- 22 个调研对象中，17 家商家明确表示，搜索广告预算中有一部分用于“拓词拿量”，占搜索预算的 12%~18%。
- 但当前医药搜索广告召回长期依赖离线关键词索引，商家即便有预算，也只能覆盖品牌词、通用名词、少量高频功效词；大量组合词、口语词、变体词、品牌/功效/人群混合表达无法稳定承接。
- 实际投放中，商家常见问题不是“愿不愿意出价”，而是“有预算但没有匹配流量可买”。从近 30 天计划日志看，医药搜索广告预算消耗率稳定在 76%~83%，说明预算并未完全跑满。
- 若平台可以提供“安全可控的扩召回能力 + query 来源解释 + 负向屏蔽能力”，11 家连锁药店和 3 个品牌愿意在试点期追加 5%~8% 搜索预算。

#### 2、医药场景不是单纯要拓量，而是要“安全前提下的确定性拓量”

- 医药商家普遍接受拓量阶段 ROI 低于核心品牌词投放，但不能接受“错配人群”“错配功效”或“高风险 query 被误召回”。
- 调研中，多数商家对扩召回的接受边界为：ROI 可接受落在 2.2~2.8 区间，但必须能够看到拓出来的 query 来自什么类型，是同义词、口语词，还是组合词，并且支持问题 query 快速拉黑。
- 尤其在儿童、孕妇、慢病、两性健康等敏感场景，商家更看重“可解释”和“可兜底”，而不是无边界放量。

#### 3、当前最大的断点不是出价能力，而是 query 理解和召回机制不足

- 现状链路是“query 命中离线关键词索引 -> 拉回广告商品”，这套机制对中高频 query 覆盖尚可，但对中长尾 query 覆盖明显不足。
- 商家即使手工维护关键词，也难以覆盖真实需求表达。调研样本中，单店人工维护的有效关键词通常不超过 600 个，但近 30 天真实成交 query 往往超过 4000 个。
- 医药 query 还存在明显的“相关性不可传递”问题：`query` 和 `改写词` 相关、`改写词` 和 `商品` 相关，并不代表 `query` 和 `商品` 最终一定相关。若平台只做一层改写、不做最终商品校验，就容易把“发烧咳嗽儿童”扩成“感冒药”甚至成人药，业务风险很高。

#### 4、调研结论

结论不是简单把关键词索引做大，而是平台需要建设一套“白盒可解释、安全可控”的医药广告扩召回体系：在保留原有精准召回的前提下，用结构化意图识别、受控改写和直接相关性校验，去承接今天被漏掉的中长尾商业需求。

### 4.2 流量空间探查

#### 1、为什么要做流量空间探查

医药搜索广告扩召回不是一个“凭感觉加召回”的项目，而是要先确认三件事：

- 有没有真实被漏掉的商业需求。
- 这些需求中，有多少是可以安全承接的。
- 召回补齐后，是否真的能转成曝光、点击和收入。

#### 2、当前 query 结构与召回现状

抽样近 30 天淘宝闪购 O2O 医药搜索流量，日均搜索请求约 235 万；剔除处方药、问诊咨询、强医疗判断和禁投表达后，可商业化请求约 169 万/日，占总请求的 72%。

在这部分可商业化请求中，query 结构与当前召回覆盖如下：

- 高频精准词占 28%，召回覆盖率 90.6%。
- 中频组合词占 33%，召回覆盖率 61.4%。
- 中长尾口语/变体/混合词占 39%，召回覆盖率仅 31.1%。
- 按结构加权后，当前整体可商业化 query 召回覆盖率约 57.8%。

这说明问题不是“整体没广告可投”，而是高频词已经基本被吃透，中长尾高意图需求仍有明显漏损。

#### 3、漏召回请求主要缺在哪里

对未召回的可商业化 query 抽样 5000 条，发现主要缺口如下：

- 品牌/功效/人群混合表达，占 29%。例如“999 儿童退烧”“女性私处止痒”。
- 组合词表达，占 27%。例如“胃胀 备点药”“旅行晕车备用”。
- 口语词表达，占 23%。例如“拉肚子药”“上火牙疼药”。
- 变体词/错拼词，占 21%。例如“布洛芬混悬夜”“念慈庵枇杷膏错拼”。

进一步分层后可看到：

- 其中约 64% 的漏召回请求，具备清晰商品意图，经过安全改写和直接相关校验后可以承接。
- 约 21% 属于意图不够稳定的灰区表达，只适合做结构化解析，不适合直接扩召回。
- 约 15% 为高风险表达，包括处方、诊断、孕妇/婴幼儿敏感用药、联合用药、剂量/副作用咨询等，不应进入广告扩召回。

#### 4、商业空间评估

- 相关资源位当前 adload 约 28.4%，仍有一定承载空间。
- 医药搜索广告日均广告 ASN 约 2.94，说明请求侧不是完全没有展示位空间。
- 医药搜索广告日均消耗约 118 万，预算消耗率 76%~83%，说明商家侧仍有增量预算可以被承接。
- 从 query 结构看，中长尾请求虽然召回弱，但意图并不低。抽样显示，这部分请求的点击率约为大盘医药广告均值的 0.92~0.98 倍，转化率约为大盘的 1.00~1.08 倍，只要不误召回，其商业价值是成立的。

#### 5、初步结论

医药搜索广告扩召回同时具备三种空间：

- 有真实需求空间：大量可商业化 query 今天没有被广告系统承接。
- 有真实预算空间：商家预算没有完全跑满，且对安全拓量有明确意愿。
- 有真实产品空间：现有系统缺的是“安全扩召回机制”，不是简单缺一个词包。

因此，该项目值得作为医药搜索广告收入的专项立项推进。

## 5. 项目目标

### 5.1 指标定义

#### 北极星指标

- 医药搜索广告收入。

#### 链路过程指标

- 可商业化 query 召回覆盖率。
- 中长尾 query 召回覆盖率。
- 新扩召回生效率：进入扩召回通路的请求中，最终能回到至少 1 个有效广告候选的占比。
- 可解释覆盖率：新扩召回曝光中，携带“改写类型 + 匹配证据 + 安全标签”的曝光占比。
- 直接相关校验通过率：扩召回候选中，通过 `query -> 商品` 最终一致性校验的占比。

#### 效率指标

- 广告 ASN。
- CTR、CVR、CXR。
- CPC、ROI。
- 新扩召回点击占比：新增通路贡献点击 / 广告总点击。

#### 风险指标

- 低相关抽样率：人工抽样中，query 与广告商品明显不相关的占比。
- 敏感 query 误召回率：高风险 query 被扩召回进入展示的占比。
- 自然 CXR 跌幅。
- 用户负反馈率、投诉率。

#### 供给健康指标

- 可售率。
- 缺货率。
- 资质通过率。
- 素材过审率。

### 5.2 目标测算

预计收入 = 原广告消耗 × （1 + 新增广告消耗占比）

新增广告消耗 = 可商业化请求量 × 新增有效召回率 × 新增广告 ASN × CTR × CPC × 安全折损

其中：

- 一期仅覆盖淘宝闪购 O2O 医药搜索广告。
- 一期仅开放 OTC、保健、器械、成人计生四类可投商品，不覆盖处方药、问诊和强诊疗表达。
- 可商业化请求量约 169 万/日。
- 新增有效召回率预计为 9%~13%。
- 新增广告 ASN 预计为 1.35。
- 新扩召回 CTR 按 5.6% 估算。
- 平均 CPC 按 2.9 元估算。
- 考虑安全过滤、排序截断和部分计费折损，安全折损按 82%~88% 估算。

测算下限：

- 新增广告消耗 = 169 × 9% × 1.35 × 5.6% × 2.9 × 82% = 2.73 万/日。

测算上限：

- 新增广告消耗 = 169 × 13% × 1.35 × 5.6% × 2.9 × 88% = 4.24 万/日。

输出结果：

- 一期预估新增广告消耗为 2.73 万~4.24 万/日。
- 对应当前医药搜索广告消耗基线 118 万/日，整体增幅约 +2.32%~+3.59%。

这个预估值偏保守，原因是：

- 只计算了可商业化且安全分层允许扩召回的 query，没有把高风险 query 计入。
- 一期只在搜索场域做扩召回，没有叠加推荐、店铺页等其他商业化增量。
- 一期只采用白盒可解释通路，不直接上线黑盒向量召回，因此覆盖提升被有意控制在安全范围内。

## 6. 项目动作

### 6.1 场景边界与 query 风险分层

#### 6.1.1 触发条件

1、触发条件

- 用户在淘宝闪购发起医药搜索请求，且请求进入广告召回阶段。

2、一期生效范围

- 资源位：淘宝闪购搜索结果页广告位。
- 类目：OTC、保健、器械、成人计生。
- 业务：仅 O2O 医药，不含 B2C、不含问诊。

3、准入规则同时满足

- 规则 1：query 通过广告可商业化判定。
- 规则 2：query 不命中禁投/高风险词表。
- 规则 3：候选商品在当前城市可售、资质合规、库存健康。
- 规则 4：候选商品具备可解释的匹配证据链。

#### 6.1.2 query 风险分层

为了避免“所有 query 都扩”，本项目先对 query 做三层风险分级：

1、L0 高风险，不扩召回

- 处方药名、诊断结果、严重疾病判断、联合用药、剂量咨询、副作用咨询。
- 孕妇、婴幼儿、慢病长期用药等高敏感场景。
- 这类请求只保留现网严格精准通路，扩召回关闭。

2、L1 低风险，允许精准结构化召回

- 通用名、品牌名、剂型、规格、标准功效词。
- 例如“布洛芬混悬液”“999 感冒灵颗粒”“润肠通便”。
- 这类请求可走精确匹配和结构化解析，但改写阈值可以相对低一些。

3、L2 中风险，允许受控扩召回

- 口语词、组合词、品牌/功效/人群混合词、场景词。
- 例如“儿童退烧备用”“胃胀反酸买点药”“女生止痒喷雾”。
- 这类请求必须同时通过改写安全校验和最终商品直接相关校验，才允许进入扩召回。

#### 6.1.3 query 结构化解析

对进入广告召回的 query 统一解析六类槽位：

- 品牌。
- 通用名。
- 功效/症状。
- 人群。
- 剂型。
- 规格。

这样做的目的不是为了“理解得更复杂”，而是为了让后续召回、排序和风险拦截都有白盒依据。

### 6.2 改写生成与治理

#### 6.2.1 改写来源

医药广告不适合在线直接做黑盒大模型改写，一期采用“离线挖掘 + 线上白盒执行”的方式，改写来源包括：

1、行业同义词和口语词词典

- 例如“拉肚子”对应“腹泻”，“上火”对应“咽喉肿痛/口腔溃疡”等标准表达。

2、错拼与变体归一

- 例如品牌错拼、通用名错拼、剂型简称、规格变体。

3、结构化拆解与重组

- 例如把“999 儿童退烧”拆成品牌 + 人群 + 功效，再映射回规范 query。

4、历史稳定成交 query 挖掘

- 只保留过去 30 天内、由精准通路已验证过的 `query -> 商品` 稳定对，用于相似 query 回放。

5、向量/大模型离线发现

- 只用于离线发现候选改写，不允许直接在线生效。
- 候选改写必须经过规则校验和运营审核后，才能进入正式改写库。

#### 6.2.2 允许与禁止的改写类型

1、允许的改写

- 别名归一：品牌简称、药品俗称、标准通用名映射。
- 口语归一：生活化表达映射到合规功效词。
- 错拼纠正：仅限高确定性的品牌/通用名错拼。
- 结构补全：品牌 + 功效 + 人群混合 query 还原为标准结构。

2、禁止的改写

- 症状泛化成更宽的药品大类。
- 儿童/成人、男性/女性、孕妇/普通人群之间的跨人群替换。
- 处方/非处方之间的跨类替换。
- 品牌 A 直接扩到竞品品牌 B。
- 涉及诊断、病情判断、副作用、联合用药的 query 扩召回。

#### 6.2.3 改写置信分

定义 `RewriteConf` 作为改写可信度：

`RewriteConf = 0.35 * 词面相似度 + 0.25 * 历史共点击稳定度 + 0.20 * 知识图谱关系分 + 0.20 * 审核稳定度`

阈值设定：

- L1 query：`RewriteConf >= 0.75` 才允许进入扩召回。
- L2 query：`RewriteConf >= 0.85` 才允许进入扩召回。
- L0 query：不允许进入扩召回。

这样设计的核心目的是：同样是改写，中风险 query 要更严格，避免“看起来像相关”但实际不安全的召回。

### 6.3 双闸门安全扩召回链路

#### 6.3.1 整体结构

整体召回结构为：

`基础精准召回（现网） + 结构化意图召回（新增） + 安全改写召回（新增） + 历史稳定 query 回放召回（新增）`

其中新增通路不是替代现网，而是“补一小批高确定性候选”。

#### 6.3.2 关键原则：相关性必须做最终闭环

本项目明确不接受“二跳相关即放量”的逻辑，而是要求每个扩召回候选都经过三段验证：

1、`query -> rewrite` 是否成立

- 看改写类型是否合法、`RewriteConf` 是否达标、query 风险等级是否允许。

2、`rewrite -> 商品` 是否成立

- 看改写词能否在广告索引中命中对应商品，且商品本身为可投、可售、合规状态。

3、`query -> 商品` 是否最终成立

- 即便前两步成立，仍然要回到原始 query，对商品做最终直接相关校验。
- 只有原始 query 和商品在品牌/通用名/功效/人群/剂型/资质上最终一致，才允许进入候选池。

#### 6.3.3 最终直接相关校验

定义 `DirectMatchScore`：

`DirectMatchScore = 0.35 * 品牌/通用名一致 + 0.25 * 功效一致 + 0.20 * 人群一致 + 0.10 * 剂型规格一致 + 0.10 * 资质风险一致`

阈值规则：

- L1 query：`DirectMatchScore >= 0.70`。
- L2 query：`DirectMatchScore >= 0.80`。
- 涉及儿童、女性、两性、慢病等敏感人群词时，人群一致必须为 1，否则直接拦截。

#### 6.3.4 在线流程

步骤 1：query 解析与风险打标

- 输出 query 类目、槽位信息、风险等级。

步骤 2：现网精准召回

- 先走原有离线关键词索引，保证高意图精准词的稳定性。

步骤 3：结构化意图召回

- 对品牌、通用名、功效、人群、剂型等结构化槽位做白盒召回。

步骤 4：安全改写召回

- 对通过 `RewriteConf` 阈值的改写词，走改写索引召回。

步骤 5：历史稳定 query 回放召回

- 仅允许使用已经被现网精准链路验证过的稳定对。

步骤 6：直接相关校验

- 对所有新增候选进行 `query -> 商品` 终审。

步骤 7：合并、去重、打标

- 为每个候选打上 `recall_source`、`rewrite_type`、`RewriteConf`、`DirectMatchScore`、`risk_level` 等标签，供排序和解释使用。

#### 6.3.5 候选配额控制

为了控制污染，一期给新增通路设置配额上限：

- `K_exact = 200`
- `K_struct = 120`
- `K_rewrite = 60`
- `K_replay = 40`
- 新增通路候选占最终粗排池的比例上限 `p_expand_cap = 25%`

这样可以保证扩召回是“补充通路”，而不是让粗排池被长尾候选塞满。

### 6.4 排序、配额与竞价保护

#### 6.4.1 排序新增特征

在粗排/精排阶段补充以下特征：

- `recall_source`
- `rewrite_type`
- `RewriteConf`
- `DirectMatchScore`
- `risk_level`
- `supply_health`
- `qualification_ok`

#### 6.4.2 排序策略

1、现网精准召回候选

- 继续沿用现网排序逻辑，避免影响高频精准 query 的既有收入。

2、扩召回候选

- 在 `Score_base` 基础上增加匹配保护和风险惩罚。

可解释表达如下：

`Score_expand = Score_base × MatchFactor × SafetyFactor`

其中：

- `MatchFactor` 由 `RewriteConf` 和 `DirectMatchScore` 共同决定。
- `SafetyFactor` 由 query 风险等级和商品资质决定，不满足条件直接为 0。

#### 6.4.3 展示位保护

为了避免低相关候选冲进高位，一期增加以下保护规则：

- 扩召回广告在 Top3 中最多占 1 个。
- L2 query 下，扩召回广告默认不得占据 Top1。
- 高风险边界 query 若无高分扩召回候选，则直接回退到精准召回，不强塞广告。
- 单请求若扩召回候选都低于精排阈值，则不额外提升 adload。

#### 6.4.4 竞价与收入保护

本项目的目标不是牺牲收入换覆盖，因此不单独给扩召回候选“流量补贴”，仍然进入统一广告拍卖池。

这样可以保证：

- 有效候选能自然转成收入。
- 低相关候选即便被召回，也会因排序和竞价不足被淘汰。
- 收入增量来自真实商业需求补齐，而不是人为让流。

### 6.5 商家侧解释能力与运营治理

#### 6.5.1 商家侧透出

为了满足“可解释”，在商家投后看板补充扩召回解释能力：

1、query 来源标签

- 精准词。
- 同义词。
- 口语词。
- 变体词。
- 组合词。

2、query 明细报表

- 展示近 7 天新增消耗 query、对应改写类型、曝光点击、成交结果。

3、负向屏蔽能力

- 商家可对问题 query 做一键屏蔽。
- 对儿童、孕妇、老人等敏感人群词支持单独屏蔽开关。

#### 6.5.2 运营侧治理

运营需要具备两类工具：

- 热点 query 审核队列：对高消耗、高投诉、高风险 query 做人工复核。
- 改写词库治理台：支持按改写类型、类目、品牌、城市做上下线和回滚。

这样做的核心是把医药扩召回从“纯算法黑盒”拉回到“平台可运营、可治理”的状态。

### 6.6 监控与回滚机制

#### 6.6.1 日常监控

上线后重点监控以下指标：

- 可商业化 query 召回覆盖率。
- 中长尾 query 召回覆盖率。
- 低相关抽样率。
- 敏感 query 误召回率。
- CTR、CVR、CXR。
- 自然 CXR 跌幅。
- 用户负反馈率、投诉率。

#### 6.6.2 红线与回滚

设置如下红线：

- 低相关抽样率 > 1.0%，立即缩量。
- 敏感 query 误召回率 > 0.3%，立即关闭对应改写类型。
- 自然 CXR 跌幅 > 0.05pp，触发全链路回看。
- 投诉率显著上升，先关 L2 query 的扩召回，再排查词库。

回滚开关按四层设置：

- 改写类型级回滚。
- 类目级回滚。
- 资源位级回滚。
- 全量开关回滚。

## 9. 项目复盘

### 9.1 项目 STAR

#### S 背景

医药搜索广告长期依赖离线关键词索引，对高频精准词覆盖较好，但对组合词、口语词、变体词和品牌/功效/人群混合表达覆盖不足，导致大量真实可商业化需求没有被广告系统承接；同时医药场景风险高，不能像普通电商一样直接做黑盒拓量。

#### T 目标

在不突破安全和用户体验红线的前提下，提升中长尾 query 的广告召回覆盖，带动医药搜索广告收入增长。

#### A 动作

1、先做 query 风险分层，明确什么能扩、什么不能扩。

2、建设白盒改写体系，只允许同义、口语归一、错拼纠正和结构补全等高确定性改写生效。

3、引入三段式校验，强制要求 `query -> rewrite -> 商品` 之外，再完成 `query -> 商品` 的最终直接相关校验，解决相关性不可传递问题。

4、在排序层增加匹配保护和展示位配额，避免扩召回候选冲击高位体验。

5、补齐商家侧解释报表和运营治理后台，让扩召回可理解、可追溯、可回滚。

#### R 结果

实验结果显示，方案在安全约束下显著提升了中长尾 query 的承接能力：

- 可商业化 query 召回覆盖率 +9.4pp。
- 中长尾 query 召回覆盖率 +17.5pp。
- 广告 ASN +0.16，广告消耗 +3.1%。
- 低相关抽样率和敏感 query 误召回率均控制在红线内。
- 推全后预估日均新增广告消耗约 +3.6 万。

### 9.2 项目优化点

优化点 1：直接相关校验从规则分升级为学习型校验器

- 当前 `DirectMatchScore` 仍以规则和打分卡为主，下一步可以升级为轻量学习模型，在保留白盒约束的前提下提升边界 query 识别能力。

优化点 2：改写发现自动化升级

- 目前向量/大模型只用于离线发现候选，后续可以建设“发现 -> 审核 -> 生效 -> 监控”的自动流水线，提升词库更新效率。

优化点 3：商家负向控制更精细

- 一期只支持 query 级屏蔽，后续可以支持按人群词、功效词、品牌词、类目词做分层屏蔽，增强商家对风险边界的掌控感。

优化点 4：扩场域复制

- 一期只覆盖搜索广告；后续可把“query 风险分层 + 白盒扩召回 + 直接相关校验”的能力复用到推荐搜索化场景和 B2C 医药广告场景中。
