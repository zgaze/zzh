# 医药 Query 改写商业化方案（产品经理视角）

## 一、先纠偏：这份材料现在更像算法周报，不像 PM 项目方案

如果站在商业化策略产品经理视角，这个项目不能讲成“我们做了多阶段 SFT + GRPO 的 Query 改写模型”，而要讲成：

- 医药搜索广告长期依赖离线关键词索引，对中高频 query 覆盖较好，但对组合词、口语词、变体词、品牌/功效/人群混合表达覆盖不足。
- 结果是大量本来有明确购买意图的 query，广告系统召不回商品，造成曝光损失和收入漏损。
- 但医药场景风险高，不能像普通电商一样直接做黑盒扩召回，必须同时解决相关性、安全性和可解释性。
- 因此平台建设了一套“基于多阶段语义理解的安全 Query 改写扩召回能力”，本质是用生成式模型补齐长尾 query 覆盖，同时保留倒排索引的高效、透明和可控。

一句话概括：

这不是一个“模型训练项目”，而是一个“医药搜索广告安全扩召回项目”。

---

## 二、正确的 STAR 写法

### S：Situation

医药搜索广告当前主要依赖离线关键词索引召回，对品牌词、通用名词、部分高频功效词覆盖较好，但大量中长尾 query 由于表达形式复杂，无法被索引有效匹配。这部分 query 以组合词、口语词、变体词和品牌/功效/人群混合表达为主，商业价值真实存在，但今天没有被广告系统有效承接，导致曝光和收入漏损。

同时，医药不同于普通电商，风险更高。Query 改写虽然可以提升召回，但存在“相关性不可传递”问题，即 Query 和改写词相关、改写词和商品相关，并不代表 Query 和商品最终相关。如果处理不好，容易引入低相关甚至错误召回，带来体验和业务风险。

### T：Task

我的任务是设计一套兼顾相关性、安全性和可解释性的 Query 改写扩召回方案，在不突破医药场景风控边界的前提下，提升中长尾 query 的广告召回能力，进而提升曝光、点击和广告收入。

### A：Action

我把项目拆成四层来推进：

1. 先定义业务边界，明确哪些 query 可以扩召回，哪些高风险 query 绝不能扩。
2. 建设 Query 改写能力，用多阶段语义理解生成适合倒排索引命中的关键词，补齐长尾 query 的覆盖。
3. 为了解决“相关性不可传递”，在改写相关性之外，增加 Query 到商品的最终相关性校验，确保不是只改得像，而是真的能召回对的商品。
4. 通过小流量 AB 测试先在非药品场景验证收益，再分场景推进扩量，并同步补齐药品场景缺失的数据链路。

### R：Result

在非药品场景 5% 流量、近两周线上实验中，广告收入提升 +4.41%，曝光 +2.36%，点击 +3.96%，CTR +1.57%，CVR 基本持平，折算推全后日均增收约 +3.06 万。说明这套方案已经验证了“安全扩召回带来真实收入增量”的业务价值。

药品场景当前尚未拿到增量曝光，不是因为方向不成立，而是离线相关性日志尚未上线，导致链路无法放量。因此项目下一步不是推翻方案，而是先扩非药、再补药品数据基建后推进二期。

## 三、项目背景

### 3.1 商业问题

当前医药搜索广告的核心问题，不是商家没有预算，也不是广告排序能力不足，而是大量 query 在召回层就漏掉了。

离线关键词索引有三个天然限制：

1. 存储和维护成本高，不可能覆盖足够多的长尾表达。
2. query 表达形式复杂，尤其在医药场景下，同一个需求会出现大量口语词、缩写词、错拼词、组合词。
3. 医药 query 风险高，不能直接用宽泛同义扩展代替精准匹配。

这会导致平台出现一个典型问题：

- 用户有明确需求。
- 商家有预算愿意投。
- 商品也真实存在。
- 但广告系统因为 query 没命中索引，导致广告完全没有被召回。

所以这个项目的本质不是“优化模型效果”，而是“补召回漏斗的最上游缺口”。

### 3.2 为什么现在必须做

原因有三点：

1. 中高频 query 基本已经被离线索引吃透，收入进一步增长越来越依赖中长尾 query 的承接。
2. 生成式模型已经具备较强的语义泛化能力，可以帮助平台把复杂 query 改写成索引可命中的关键词。
3. 平台已经通过 5% 非药流量实验验证了商业收益，因此当前节点具备从“技术验证”走向“产品化扩量”的条件。

### 3.3 当前验证结论

从现有实验结果看，核心结论非常明确：

- 方向有效：非药品场景已带来显著收入提升。
- 风险可控：CTR 上升、CVR 基本稳定，说明新增召回不是粗暴放量。
- 仍有分场景差异：不同 slot 效果差异明显，后续扩量不能一刀切。
- 药品场景尚未起量：根因是数据基础设施不完整，而不是模型本身无效。

## 五、核心方案设计

### 5.1 方案定位：不是替代索引，而是增强索引

这个方案不能讲成“让生成式模型直接召回商品”，而应该讲成：

平台通过 Query 改写，把用户原始 query 转成索引中已有的商品关键词，再去命中原有倒排索引。

这样做有三层好处：

1. 利用生成式模型的语义理解能力，补齐长尾 query 覆盖。
2. 继续复用现有倒排索引体系，工程成本低、响应快。
3. 保留稀疏检索的透明性和可解释性，符合医药场景安全要求。

### 5.2 产品能力拆解

从 PM 视角，这个项目不是一个模型，而是四个能力模块：

#### 能力 1：Query 语义理解

系统先识别 query 的语义类型和意图强弱，包括：

- 精搜词。
- 泛搜词。
- 较泛属性词。
- 同近义词。
- 品牌/功效/人群复合词。

这一步的意义是给后面的改写和风险控制打基础。

#### 能力 2：安全改写生成

系统基于多阶段语义理解，为原始 query 生成一组适合命中广告关键词索引的 terms。

这里的 PM 重点不是讲 SFT 或 GRPO，而是要讲清楚产品价值：

- 让复杂 query 被系统“听懂”。
- 让听懂后的结果能“命中索引”。
- 让命中后的结果仍然“可解释、可回溯”。

#### 能力 3：最终相关性校验

这是整个项目最关键的产品设计，也是和普通电商最大的不同。

不能只校验：

- Query 和改写词是否相关。
- 改写词和商品是否相关。

还必须额外校验：

- 原始 Query 和最终商品是否真的相关。

因为在医药场景下，相关性不可传递。

典型例子：

- Query：“清肺止咳颗粒”
- 改写词：“清肺止咳”
- 商品：“小儿清肺化痰泡腾片”

前两段看起来都相关，但最终商品和原始 query 并不一致，尤其在剂型、人群、规格上存在明显偏差，这类召回在医药里不能放过。

所以项目必须从产品规则上明确：

只有“词词相关 + 词品相关 + Query 到商品最终相关”三层都成立，才允许进入广告召回。

#### 能力 4：分场景灰度和可解释治理

医药扩召回不能一次性全量放开，必须分场景治理：

- 非药品先验证收益。
- 药品在日志链路补齐后再上。
- 不同资源位按效果分层放量。
- 对错误改写和 badcase 保留人工回收与回滚能力。

这说明它不是单一算法能力，而是一个完整的“产品化可控扩召回系统”。

## 六、为什么这套技术路线对 PM 是成立的

### 6.1 为什么不用纯向量召回或黑盒检索

因为医药场景最怕三件事：

- 不可解释。
- 不可控。
- 出现幻觉或违规召回。

纯向量召回或黑盒检索虽然覆盖更广，但在医药广告场景中，平台需要能回答两个问题：

1. 这个 query 为什么会召回这个商品？
2. 如果召错了，能不能快速定位是哪一步出了问题？

而 Query 改写后再命中倒排索引，本质上是把黑盒能力前置，用白盒检索承接，比较符合商业化上线要求。

### 6.2 为什么要做多任务理解，而不是只做改写

因为改写任务本身很容易只学到“召回更多词”，却学不会“哪些词不该改”。

从 PM 视角，这五类辅助任务其实分别对应不同产品能力：

- 词词相关性：约束扩召回边界。
- Query 最相关关键词：提升命中率。
- 同义词生成：补齐标准表达。
- 意图识别：识别 query 宽泛程度。
- 词品相关性：连接商品召回结果。

所以多任务学习本质上是在让模型同时学会“召回能力”和“风险意识”。

### 6.3 为什么强化学习阶段要重点加强相关性而不是 CTR

技术文档里的结论已经很清楚：

- 相关性信号有效。
- 惩罚机制进一步提升质量。
- CTR 信号反而拖累了整体表现。

从 PM 视角，这背后的业务原因是：

- CTR 不是纯粹的相关性信号，受品类、药/非药、B2C/O2O、供给质量、价格力、资源位等多因素影响。
- 如果直接用粗糙 CTR 反推改写好坏，容易把“点击高但不严谨”的词误当成优质改写。
- 医药广告更重要的是先确保“召回正确”，再追求“点击更高”。

因此，一期产品策略上应坚持“相关性优先、CTR 次之”。

## 七、实验结果的产品化解读

### 7.1 整体实验结论

非药品近两周线上 5% 流量实验结果如下：

- 收入 +4.41%
- 曝光 +2.36%
- 点击 +3.96%
- CTR +1.57%
- CVR -0.06%
- CPC +0.42%
- 提单数 +3.90%
- PVR +11%

折算推全收入约 +3.06 万/日。

这组结果对 PM 来说有三个重要含义：

1. 新增召回不是纯垃圾流量

- 如果是低质放量，通常会看到曝光涨、点击不涨，或者 CTR 显著下滑。
- 但当前 CTR 是提升的，说明改写后召回到的流量整体相关性是成立的。

2. 收益主要来自“新增有效供给进入拍卖池”

- 曝光、点击、提单数同步增长，说明改写能力真实补齐了此前召不回的 query。

3. 当前仍处于“安全收益区”

- CVR 基本持平，只下降 -0.06%，说明项目尚未明显透支转化质量。

### 7.2 分 slot 解读

分资源位数据说明后续不能做统一扩量，而要“按位点管理”：

#### 205 slot

- 收入 +10.38%
- 曝光 +4.90%
- 点击 +6.85%
- CTR +1.87%
- CVR +1.85%

结论：是最优扩量位点，优先扩大。

#### 331 slot

- 收入 +7.28%
- 曝光 +8.57%
- 点击 +5.51%
- CTR -2.82%
- CVR -6.43%

结论：虽然收入增长，但效率承压，说明该资源位对新增召回更敏感，扩量时需更严格阈值，甚至可以先不跟随主方案同步放量。

#### 37 slot

- 收入 +5.25%
- 曝光 -0.09%
- 点击 +6.00%
- CTR +6.10%

结论：典型的“提质型收益”，说明不是靠多放量，而是靠更精准的匹配提升点击效率，可以继续放。

#### 444 / 460 slot

- 收入分别 +4.11%、+3.99%
- CTR 和提单表现整体稳定

结论：属于可跟随扩量位点。

### 7.3 药品场景为什么没起量

这部分一定要从产品角度解释，不要只写“日志没上”。

正确说法应该是：

药品场景由于离线相关性日志链路尚未上线，平台当前缺乏足够可靠的 Query 到商品反馈数据，导致改写结果无法在药品链路中完成稳定放量。因此当前药品未见增量曝光，反映的是数据基础设施缺口，而不是项目方向无效。

这意味着：

- 非药品已经完成“商业价值验证”。
- 药品仍处于“数据链路待打通”的产品准备阶段。

## 八、项目动作

### 8.1 搭建基于 SFT 到生成式 Query 改写的安全扩召回底座

当前医药搜索广告主要依赖离线关键词索引召回，对品牌词、通用名词和部分高频功效词覆盖较好，但对组合词、口语词、变体词、品牌/功效/人群混合表达覆盖不足。若继续依赖规则词包和人工扩词，维护成本高、覆盖效率低，已经很难支撑中长尾 query 的进一步增长；若直接采用黑盒向量召回或生成式检索，虽然覆盖可能更广，但在医药场景下会带来可解释性差、风控边界不清、问题难回溯等缺点，不适合作为一期商业化上线方案。

因此，一期在技术路线上的产品判断是：以 `SFT 打底 + 多任务对齐 + 强化优化 + 生成式 Query 改写到 term` 作为主链路。这里的核心不是让模型直接生成商品，也不是替代现有倒排索引，而是让模型把用户原始 query 改写成能够命中现有关键词索引的 term，用生成式能力补齐语义覆盖，再由现有倒排索引承接效率、透明性和可控性。

具体做法上，先通过 Query2terms 的 SFT 训练，让模型具备基础改写能力，能够把 query 转成索引可识别的关键词；再通过词词相关性、同义词生成、意图识别、词品相关性等多任务训练，让模型不只是“会改”，还知道“哪些词该改、哪些词不能改”；最后在此基础上引入相关性奖励和惩罚机制做强化优化，进一步压低低质、泛化过度和不安全改写。产品侧对这条路线的判断是：一期先追求“高相关、可解释、可控放量”，而不是追求黑盒大幅召回。

为了让这条链路可落地，一期同步建设四个底座能力：

- `Query 风险分层`：将 query 分为低风险可扩、中风险受控扩、高风险禁扩三层，先明确什么能放、什么不能放。
- `Query 语义解析`：识别品牌、通用名、功效、症状、人群、剂型、规格，为后续改写和校验提供结构化输入。
- `改写类型分层`：将改写拆成同义词、口语词、错拼词、组合词、品牌/功效/人群混合词，不同类型配置不同阈值。
- `改写触发规则`：原 query 已能稳定召回的，不强制改写；仅对召回弱但意图清晰的 query 触发生成式改写。

这一动作解决的是：平台先拥有一个“能补长尾、又不至于失控”的生成式扩召回底座。

### 8.2 建立人工审核标注与三段式评估体系

医药场景下，词品相关性标准无法直接迁移到 `Query -> 改写词 -> 商品` 全链路，因为 `A≈B 且 B≈C` 并不代表 `A≈C`。这也是当前 Query 改写最大风险所在。要让生成式改写真正从技术能力变成可上线的产品能力，必须建立人工审核标注与三段式评估体系，而不能只看模型生成词是否“像”，还要看最终召回商品是否“对”。

因此，一期统一将标注对象定义为三元组：`Query - 改写词 - 商品/SPU`。每条样本同时标注以下信息：

- `Query 类型`：精搜词、泛搜词、属性词、混合词。
- `风险等级`：低风险、中风险、高风险。
- `Query -> 改写词` 是否合理。
- `改写词 -> 商品词` 是否合理。
- `Query -> 最终商品` 是否最终成立。
- `错误原因`：品牌偏移、剂型不一致、规格冲突、人群不一致、属性泛化、功效偏移等。

标注方式上，建议采用“样本采集 + 初审 + 复审 + 仲裁 + 质检”的流程：

- 样本采集：优先抽取高频新增 query、高消耗 query、模型新生成词、投诉 query、模型分歧样本、未生成但高价值 query。
- 初审：由训练后的运营标注同学完成，保证样本量和迭代效率。
- 复审：对药品、高风险类目和争议样本，由医药运营或具备专业背景的同学复核。
- 仲裁：对多方意见不一致的样本，按周集中仲裁，统一口径。
- 质检：抽取固定比例样本回查，确保标注一致性。

在评估规则上，正式建立三段式校验：

1. `Query -> 改写词` 相关性校验，判断生成是否合理。
2. `改写词 -> 商品关键词` 匹配校验，判断能否命中正确索引。
3. `Query -> 最终商品` 一致性校验，判断最终召回商品是否真的符合原始需求。

第三步是一票否决项。只要发现剂型不一致、规格核心冲突、适用人群冲突、品牌明显偏移、医药属性边界冲突，就不允许进入线上召回。这样做的价值在于：把原本难以讲清楚的“相关性不可传递”问题，变成一套可以审核、可以评估、可以回滚的业务机制。

### 8.3 分场景灰度放量，优先承接非药增量收入

当前线上实验已经证明，非药场景的 Query 改写扩召回具备明确商业价值，因此第三个动作的重点不再是继续讲模型，而是讲“如何在可控前提下把收益放大”。一期的放量策略建议分两层推进。

第一层是`分类目放量`。一期优先覆盖非药品场景，包括保健品、器械、成人用品等相对低风险类目。原因很直接：这部分场景已经在 5% 流量上验证了 `收入 +4.41%`、`点击 +3.96%`、`CTR +1.57%`，推全折算约 `+3.06 万/日`，说明方向成立，具备继续扩量价值。药品场景当前没有增量曝光，核心原因是离线相关性日志尚未上线，数据链路仍不完整，因此药品不作为一期放量重点，而是作为二期补齐基建后的扩量对象。

第二层是`分 slot 放量`。从现有结果看，不同资源位表现差异明显，因此不能简单按总收益统一扩量，而要按位点精细管理。205、37、444、460 等表现稳定的 slot 可以优先扩量；331 虽然收入增长，但 CTR、CVR 承压，说明该位点对新增召回更敏感，应配置更严格的阈值，甚至在一期先维持观察。产品侧需要明确一条原则：生成式 Query 改写不是“一次开全”，而是“优先承接高收益、低风险位点”，逐步放大。

为保证扩量过程可控，一期需要同步建设 `改写词审核后台` 作为放量控制面板。后台支持按类目、slot、改写类型、模型版本四个维度开关和回退，让灰度扩量从单纯算法实验，升级为产品可控的流量经营能力。这个动作解决的是：先把已经验证有效的非药收入稳定做出来，再为药品二期扩量留出空间。

### 8.4 建设改写词审核后台与 badcase 闭环治理机制

医药 Query 改写不是一个一次性上线项目，而是一套需要长期治理的系统能力。因此，必须建立以 `改写词审核后台 + badcase 闭环治理` 为核心的运营体系，让平台不仅能放量，还能知道哪里出错、如何回滚、怎样持续迭代。

其中，`改写词审核后台` 主要承接日常治理动作：

- 查看高频新增改写词及其带来的曝光、点击、提单。
- 审核某类改写词是否继续生效。
- 对问题词做下线、拉黑或限流处理。
- 按类目、slot、模型版本快速回滚。

`badcase 闭环治理` 主要承接持续优化动作：

- 建立统一 badcase 池，沉淀低相关、跨剂型、跨人群、品牌偏移、属性泛化等错误案例。
- 每周输出新增收益词、风险词、无效词、争议词四类清单。
- 将 badcase 同步回流到规则库、标注库和训练集，持续优化下一版模型。
- 对高投诉、高风险、高争议 query 建立专项巡检机制。

从产品视角看，这个动作的真正价值是把 Query 改写从“模型能力”沉淀成“平台能力”。平台不只是能依靠生成式能力拿到增量收入，还能解释为什么放量、知道哪里错了、并且有机制把错误持续修正。

## 九、分阶段推进策略

### 阶段 1：非药 5% -> 10% 扩量

建议审批思路：

- 可以支持非药扩量 10%，但不是无条件全 slot 同步扩。
- 建议按资源位分层扩量，优先放大效果稳定的 slot。
- 同时加挂 CTR/CVR/投诉率护栏，确保不会因为扩量带来质量回撤。

建议护栏：

- CTR 不得连续显著下滑。
- CVR 波动不得超过设定阈值。
- 提单数保持正向。
- badcase 和投诉率无明显抬升。

### 阶段 2：药品场景补链路

当前最关键动作不是继续调模型，而是补齐数据基础设施：

- 上线离线相关性日志。
- 建立药品场景可用的 Query-item 反馈闭环。
- 形成药品侧可灰度的最终相关性判定方案。

没有这一步，药品场景很难真正进入商业化扩量。

### 阶段 3：能力升级

后续可以继续迭代三类方向：

1. 多任务学习升级

- 让模型更稳定理解医药 query 语义和安全边界。

2. 检索方式升级

- 对 40% 仍无法有效改写的 query，考虑改写分词、词表限制、向量检索等补充方案。

3. 反馈信号升级

- CTR 不做统一口径，而按药/非药、B2C/O2O、类目、资源位分场景建模。

可以直接用下面这段：

“这个项目表面上看是一个 Query 改写模型优化，但我当时是把它当成医药搜索广告安全扩召回来做的。业务问题是广告召回长期依赖离线关键词索引，中高频 query 还能覆盖，但大量组合词、口语词、变体词召不回来，导致真实商业需求没有被承接。难点在于医药场景不能粗暴拓量，因为 Query 改写存在相关性不可传递问题。  
所以我会把方案拆成四层：先定义高风险边界，再做 Query 语义理解和改写生成，然后增加 Query 到商品的最终相关性校验，最后通过非药先灰度、药品补日志后再上。  
最终非药 5% 流量实验，收入提升 4.41%，CTR 提升 1.57%，折算推全日均增收约 3.06 万，证明这套方案不是单纯多召回，而是在风险可控前提下带来了真实商业增量。”

## 十一、最终结论

如果你从 PM 视角来讲，这个项目最应该突出的是三点：

1. 它解决的是广告召回漏损，不是单纯算法效果提升。
2. 它的核心创新不是“用了强化学习”，而是“在医药场景下把生成式改写做成了可控、可解释、可上线的商业化能力”。
3. 它的结果不是“离线指标更好了”，而是“非药品线上已验证收入增量，下一步具备扩量价值，药品场景则进入数据基建补齐阶段”。

这三点讲清楚，你的项目就会从“算法同学做了个模型”变成“商业化策略 PM 推动了一个能带收入的安全扩召回项目”。
